# PROJETO: BOLSA VIRTUAL DE EMPRESAS FICTÍCIAS

## 1. Visão geral

Criar um site de simulação econômica onde jogadores podem criar empresas fictícias, emitir cotas virtuais e negociar essas cotas dentro de uma bolsa virtual usando uma moeda interna fictícia.

O sistema deve funcionar como um jogo de economia, com login, carteira, moeda virtual, empresas criadas por jogadores, compra e venda de cotas, gráfico de preço, livro de ofertas, painel administrativo, corretores virtuais e auditoria completa.

Não deve existir dinheiro real, saque real, promessa de lucro real ou investimento real.

## 2. Objetivo do sistema

O objetivo é criar uma experiência parecida com uma bolsa de valores fictícia para comunidade, servidor RP ou jogo online.

Os usuários devem poder:

- criar conta;
- receber moedas virtuais;
- comprar cotas virtuais de empresas fictícias;
- vender cotas virtuais;
- criar empresas fictícias;
- acompanhar gráficos;
- abrir ofertas de compra;
- abrir ofertas de venda;
- acompanhar patrimônio fictício;
- ver histórico de transações.

A administração deve poder:

- criar moeda virtual;
- distribuir moeda para corretores;
- controlar a tesouraria central;
- aprovar empresas;
- suspender empresas;
- bloquear usuários;
- auditar transações;
- ver logs completos;
- controlar permissões;
- investigar movimentações suspeitas.

## 4. Perfis de usuário

O sistema deve ter os seguintes cargos:

### 4.1 Usuário comum

Pode:

- acessar painel pessoal;
- ver saldo;
- comprar cotas virtuais;
- vender cotas virtuais;
- abrir ordens de compra;
- abrir ordens de venda;
- cancelar ordens próprias;
- ver empresas listadas;
- criar solicitação de empresa fictícia;
- ver histórico da própria carteira.

### 4.2 Empresário

É um usuário que possui empresa aprovada.

Pode:

- editar descrição da própria empresa;
- publicar comunicados oficiais da empresa;
- solicitar nova emissão de cotas;
- acompanhar investidores da empresa;
- acompanhar gráfico da empresa.

Não pode:

- alterar preço manualmente;
- criar cotas infinitas;
- apagar histórico;
- mexer em saldo de usuários.

### 4.3 Corretor virtual

É um membro autorizado pela administração para distribuir moeda virtual aos usuários.

Pode:

- receber moeda da Tesouraria Central;
- repassar moeda virtual para usuários;
- ver histórico dos repasses que ele fez;
- justificar cada repasse;
- consultar usuários atendidos por ele.

Não pode:

- criar moeda;
- apagar registros;
- alterar saldo manualmente sem registro;
- alterar preço das cotas;
- aprovar empresas;
- manipular mercado.

Todo repasse feito por corretor deve gerar log permanente.

### 4.4 Auditor

Pode:

- ver todos os logs;
- ver transações;
- ver repasses;
- ver emissões de moeda;
- ver movimentações suspeitas;
- exportar relatórios.

Não pode:

- alterar saldo;
- criar moeda;
- aprovar empresas;
- bloquear usuários;
- apagar dados.

### 4.5 Administrador

Pode:

- aprovar empresas;
- suspender empresas;
- bloquear usuários;
- desbloquear usuários;
- ver painel geral;
- gerenciar usuários;
- gerenciar empresas;
- gerenciar ordens;
- investigar denúncias.

Não pode criar moeda se não for ADM Chefe da Moeda.

### 4.6 ADM Chefe da Moeda

É o cargo equivalente ao Banco Central do jogo.

Pode:

- criar moeda virtual;
- enviar moeda para a Tesouraria Central;
- enviar moeda para corretores;
- definir limites de emissão;
- configurar taxas;
- controlar inflação;
- revisar grandes movimentações;
- congelar contas suspeitas;
- aprovar ou negar geração de moeda;
- visualizar todos os registros econômicos.

Esse cargo deve ser extremamente limitado.

### 4.7 Super Admin

É o dono técnico do sistema.

Pode:

- gerenciar permissões;
- gerenciar cargos;
- acessar configurações gerais;
- corrigir erros críticos;
- executar manutenção.

Mesmo o Super Admin não deve apagar logs financeiros. Correções devem ser feitas por estorno registrado.

## 5. Fluxo da moeda virtual

A moeda deve seguir este fluxo:

ADM Chefe da Moeda
↓
Tesouraria Central
↓
Corretores virtuais
↓
Usuários
↓
Compra e venda de cotas virtuais

A moeda nunca deve aparecer sem registro.

Toda criação de moeda precisa registrar:

- quem criou;
- quantidade;
- data;
- motivo;
- destino;
- saldo anterior;
- saldo posterior.

Todo repasse para corretor precisa registrar:

- quem enviou;
- quem recebeu;
- quantidade;
- data;
- motivo.

Todo repasse para usuário precisa registrar:

- corretor responsável;
- usuário beneficiado;
- quantidade;
- data;
- motivo;
- saldo anterior;
- saldo posterior.

Regra econômica adicional da bolsa fictícia:
- toda taxa de compra/venda deve ser dividida em 50% para a plataforma e 50% para a empresa negociada;
- receita da empresa acumula primeiro na carteira da própria empresa (não vai direto para o dono);
- sem retirada real, sem saque real, sem blockchain, sem cripto.

## 13. Motor de negociação

Criar um matching engine simples.

Regras:

- uma ordem de compra deve cruzar com a menor ordem de venda compatível;
- uma ordem de venda deve cruzar com a maior ordem de compra compatível;
- ordens mais antigas têm prioridade quando o preço é igual;
- permitir execução parcial;
- bloquear saldo quando usuário abre ordem de compra;
- bloquear cotas quando usuário abre ordem de venda;
- liberar saldo/cotas quando ordem for cancelada;
- gerar trade quando compra e venda forem executadas;
- atualizar preço atual da empresa com base no último trade;
- atualizar gráfico com base nos trades.

## 20. FASE 4A IMPLEMENTADA — LIVRO DE OFERTAS E MATCHING ENGINE SIMPLES

A Fase 4A adiciona negociação entre usuários no mercado secundário com regras de proteção e auditoria.

### 20.1 Tipos de ordem

- Ordem limitada de compra: bloqueia saldo e entra no livro de compras.
- Ordem limitada de venda: bloqueia cotas e entra no livro de vendas.
- Compra a mercado: consome as menores ofertas de venda, permite parcial e respeita slippage.
- Venda a mercado: consome as maiores ofertas de compra, permite parcial e respeita slippage.

### 20.2 Regras econômicas aplicadas

- Nunca permite saldo negativo ou cotas negativas.
- Empresa inexistente ou suspensa não pode ser negociada.
- Ordem parcialmente executada permanece com quantidade restante.
- Ordem totalmente executada vira FILLED.
- Cancelamento libera saldo/cotas bloqueadas.
- Cada execução gera Trade + Transaction + CompanyOperation + AdminLog.
- Preço atual da empresa é atualizado pelo último trade.
- Taxa de compra é aplicada ao comprador e taxa de venda ao vendedor.

### 20.3 Banco de dados

Foi adicionado o modelo `Trade` com:
- companyId, buyerId, sellerId;
- buyOrderId, sellOrderId;
- quantity, unitPrice, grossAmount;
- buyFeeAmount, sellFeeAmount;
- createdAt.

Também foram adicionados campos em `MarketOrder` para controle de execução e bloqueio:
- remainingQuantity;
- lockedCash;
- lockedShares;
- updatedAt;
- canceledAt.

### 20.4 Endpoints do mercado

- `POST /api/market/orders`
- `GET /api/market/orders/me`
- `GET /api/market/companies/:companyId/order-book`
- `POST /api/market/orders/:id/cancel`
- `POST /api/market/companies/:companyId/buy-market`
- `POST /api/market/companies/:companyId/sell-market`
- `GET /api/market/companies/:companyId/trades`

### 20.5 Slippage

Slippage é a tolerância máxima de variação de preço para ordens a mercado.

Exemplo simples:
- melhor venda atual = 10;
- slippage = 5%;
- compra a mercado aceita até 10,50.

Se o livro exigir preço acima da tolerância, a execução é bloqueada.

### 20.6 Escopo não implementado nesta fase

- gráfico candlestick;
- motor avançado de candles;
- qualquer operação com dinheiro real, saque real ou cripto real.
